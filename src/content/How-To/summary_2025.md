###### 今年寫業務邏輯為重，關鍵字抓對技術上不會有問題，有更多時間能思考如何寫出好維護的程式碼，這邊小結一下今年所學的幾個觀念。

---

### 虛實物件分離

以發票為例，狀態應僅有四種：未開立、已開立、已折讓、已作廢；這些能被**觀察**的狀態。那從「發票作廢」開始延伸，

以前需拿著 ***實體發票*** 在時限內繳回政府，實體流轉本身就是進度的追蹤依據：

1. 發票在買方手上 → 交易尚未取消
2. 發票已繳回賣方 → 雙方同意，賣方可申報
3. 發票隨申報表送出 → 向稅務機關申報作廢
4. 稅務機關已收存 → 作廢完成

但在電子化時代達成「已作廢」的過程變得複雜：

1. 內部階段：客服受理 > 客戶簽名 > 會計審核
2. 外部階段：平台傳輸 > 財政部驗證 > 等候結果
3. 終點：財政部核可 → 觸發事件，將狀態改為「已作廢」。

上述流程會衍生出 **不能被觀察** 的狀態，例如待處理、待上傳等等 ...

若能清楚分離「實體狀態」與「流程狀態」就可以設計為：

- **主表**：記錄發票的實體狀態（未開立 / 已開立 / 已折讓 / 已作廢）← 對應稅務規定
- **子表**：記錄作廢流程狀態（State 1~9）← 對應內部流程
    - State 1~4：內部審核（受理、簽名、核對...）
    - State 5~8：與財政部往來（上傳、驗證、查詢...）
    - State 9：財政部核准 → 觸發同步，更新主表為「已作廢」

主表保持簡潔子表完整追蹤進度。這是閱讀 Domain-Driven Design 後的小啟發，就算沒有用到架構也是以融入現有的業務邏輯。

### 彈性與異常處理

當新項目開發到一半，發現必須改舊有程式碼才能避免 0.02% 的重複紀錄，兩個選項：

1. 花一個月，改舊程式讓資料完全正確
2. 花五個工作天，額外寫定時流程去補償

市場上品質好、價格低、供貨穩定三者不可兼得，在軟體工程領域也是如此；要如何處理現在決於業務性質：

- **金流系統**（牽涉商譽、三方資料）→ 選方案一，確保一致性
- **記帳對帳**（允許延遲）→ 選方案二，開發快、可維護性高

只要能明確定義「異常」和後續處理方式，它就成為可控的預期行為，而非系統錯誤。根據風險等級設計容錯策略，不需要每處都追求零錯誤。

### 效能提升

初學時容易被效能問題卡住，糾結 `Array vs Set`、React 要不要包 `memo()`、`state & debounce` 等等。但其實寫高階語言或框架時，這些優化是讓程式從 80 分到 88 分的微幅提升，相較之下維護性與可讀性更重要。例如：

- 用 `Array` 而非 `List` 是為了防止不小心添加元素
- 整理成 `Map` 而非 `list.find()` 是為了清楚表達意圖、防呆

真正要有感的效能提升，會是來自架構層面的思考：

- 查詢邏輯要在後端還是資料庫？有加 Index 嗎？
- 資料可以讓客戶端暫存嗎？封包大小符合建議值嗎？
- 花兩個月優化程式碼，還是評估橫向擴展或升級硬體更划算？

能從源頭修改，效能提升最多。工程師的職責是選擇並善用現有的成熟工具，而非花時間最大化每個細節。（延伸閱讀：[細節不決定成敗](https://youtu.be/HnwJZjEhrf8)）
